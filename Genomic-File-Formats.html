Genomic File
Formats

BED file format

There are numerous types of files used for storing and accessing genomic data. One of the most common file types is the Browser Extensible Data (BED) file type. The BED file is designed to store information on genomic intervals through tabulated columns, such as a feature for an exon with the starting and ending position. It has at the minimum three columns: the chromosome, position start, and position end. Since BED files contain genomic positions, one of the first steps of caution for a bioinformatician is to understand if the positions in the BED file as based on a 0-based or 1-based interval. The difference between which interval is used can affect the browser, albeit one will not know until it is compared to a reference and the intervals are shifted by one position. For example, the UCSC Genome Browser uses a 0-based interval. Large versions of the BED file type (named BED-12) contains up to 12 columns, which include the strand, score of the interval (relative to certain calculations), name, etc. The caveat here is that columns go in sequential order based on the column type and cannot be skipped. For example, you cannot go from column 3 to column 6 without column 4 and column 5. For more specific information on BED file formats, visit this link, which also contains information about other bioinformatic file types.   

WIG and BigWig File Format

Another common file type is the wiggle (WIG) file type. The WIG file is designed to store continuous data at each genomic location, such as chromosomal modifications with locations, across a large genomic region. In the same as BED files, WIG files can be used with genome browsers, but will require header lines that contain metadata for the browser to interpret specific locations. An alternate version of the WIG file format is the BigWig format, which is a compressed version of the WIG format that omits non-relevant information for genome browsers to load the display faster. While BIG and WIG, as well as BigWig, file formats are used for intervals on genomic locations, we will need something else to deal with larger amounts of data- such as that of an entire aligned, sequenced genome. The two most common file formats used are the SAM and BAM files formats. 

SAM File Format

The Sequence Alignment/Map (SAM) file format came to be used during the 1000 Genomes Project. The challenge with the 1000 Genomes Project is that it needs a file format to hold not only the sequence and interval locations, but also the alignment of it to a reference genome after the sequence is aligned. SAM file formats are generally the result of aligning next-generation sequencing (NGS) from ChIP-seq or RNA-seq, using alignment programs, such as HISAT2 or BWA. In addition to location mapping, SAM file formats store information about how the sequence aligns to the reference genome with 11 mandatory fields columns: QNAME, FLAG, RNAME, POS, MAPQ, CIGAR, MRNM, MPOS, ISIZE, SEQ, and QUAL. Each field gives information on the sequence in mapping to the reference genome. For example, the CIGAR field is a string of symbols that encoding the variations in positions of the alignment between the read sequence (the queried sequence) and the reference genome. Within the CIGAR string, a M indicate that position was a match, a I indicate the position had an insertion to the reference genome, and a D indicate the position had a deletion from the reference genome. BAM File Format

Another common file format for alignment results is the Binary Alignment Map (BAM) file format. In actuality, a BAM file format contains the same information as the SAM file format. However, the actual data within the BAM file format is compressed and encoded into a binary form. The benefit of a BAM file is that compression reduces the file size (in bytes) to be nearly equivalent to the number of base-pairs in the sequenced file. The compression also allows for genomic browsers to retrieve and display data at a faster rate, when it is sorted. On the topic of sorting, BAM files need to be sorted in their intervals in relationship to the reference genome. The BAMtools suite in Galaxy automatically sorts BAM file outputs, but there is also the tool command sort to sort BAM files if necessary. This step would be critical when manipulating BAM files within a Unix shell. In addition to sorting, the BAM files require a corresponding BAM-Indexed (BAI) file with the same name when displaying the BAM file on a genomic browser. The BAI file format is used by the browser, in the background, to correctly place the intervals based on the indexing.   


BEDtools Suite

It is one thing to know the options of file formats- it is another to know how to manipulate file formats to obtain the needed data. We will begin with two popular suite of tools that manipulate data, both of which are found amongst Galaxy’s list of tools to utilize for bioinformaticians. The first suite of tools is the BEDtools suite, which is widely known as the “Swiss Army Knife” of manipulating genomic data for genomic intervals. Outside of Galaxy, the BEDtools suite can be installed on a Unix server and commands can be entered through a Unix shell to perform operations.   

In addition to the figure below that contains a list of commands for BEDtools, the command bedtools -help will also display current command options for the installed version. The general syntax of the commands on Unix will be performed in this manner: bedtools <command> [command options]. Commands, such as intersect, map, or sort; can employ multiple options for the desired output. For example, the command intersect is a tool used to find overlaps in the intervals of features within different BED files. Using this example, the option -wa is used to obtain the full interval of file A that overlaps with any part of file B. The option -v is used to obtain intervals from file A that does not intersect with any intervals on file B. SAMtools Suite

The second suite of tools is the SAMtools suite, which is also available on Galaxy or can be locally installed into a Unix server. The command tools within SAMtools allow for manipulation of SAM file formats, as well as other alignment-based file formats (BAM or CRAM). For example, the view command allows for conversion of SAM file formats to BAM file formats. If performing operations with SAMtools within a Unix shell the command is: samtools view -bS filename.sam > filename.bam. The -b option specifies the output as a BAM file and the -S command specifies the input as a SAM file. SAMtools also has the ability to perform operations on BAM files, such as the sort or index commands that, respectively, sort or index the BAM file inputted to give the desired output. Note that the order of which options go first is not necessary in Unix. For more information on performing commands through a Unix shell, visit this page for a broad overview of Unix command syntax. If using Galaxy, keep in mind that, similar to BAMtools, there is not an explicit index command within this suite as Galaxy automatically runs the index command as part of the pipeline to generate the BAI file for other tools.   

Exercise Practice with BEDtools in Galaxy

For this practice, we will perform a longer exercise. The tools you will need are two bed files and the Galaxy server. The first BED file contains intervals of exons on the q31 region of chromosome 5. The second BED file contains intervals of CpG regions in that same region on chromosome 5. Using the Galaxy server, upload both files as a BED file type and leave all other fields as default. Within Galaxy, find the BEDtools Intersect Interval tool (under Genomic File Manipulation > BED) on the left-hand column list. Alternatively, you can also search for “BEDtools intersect interval”. Within BEDtools, you will need to start with having the File A option selected for hs_5q31_exons.bed and the File B option selected for hs_5q31_cpg.bed. This means that our file to use is the BED file with exon intervals and the file we are comparing to is the file of CpG intervals. Now, we will answer the following four questions:   

1.       How many unique exons intersect a CpG region? 

2.       How many unique exons do not intersect CpG regions? 

3.       How many unique CpG regions intersect an exon? 

4.       How many unique CpG regions do not intersect exons?   

First Question: For this part, the goal is to find unique exons that intersect a CpG region. Thus, the (-u) option in the parameters will be selected for to show only unique exons intersected. All other parameters are held at default. The result showed that there were 542 unique exons that intersected a CpG region. Your result should be the same as the figure below.   

Second Question: Using the same tool in a separate instance, the option for (-v) will need to be selected for in order to obtain exons that do not intersect with a CpG region. All other parameters are held at default. The result showed that 4,296 exons did not intersect with a CpG region. Your result should be the same as the figure below.  

Third Question: Compared to the first and second questions, we have File A set as the exon intervals file and File B set as the CpG intervals file. However, this question will not focus on the CpG intervals file and compare that to the exon intervals file. Keep in mind that the directionality does matter! In order to do this, we will need to set File A for hs_5q31_cpg.bed and File B for hs_5q31_exons.bed. Using the same tool in a separate instance, the (-u) option in the parameters is selected for to show only unique CpG regions that intersected. All other parameters are held at default. The result showed that there were 200 unique CpG regions that intersected an exon. Your result should be the same as the figure below.  

Fourth Question: Set the file options in the same manner as the third question. Using the same tool in a separate instance, the (-v) option in the parameters is selected for to show only unique CpG regions that did not intersect with an exon. All other parameters are held at default. The result showed that there were 45 unique CpG regions that did not intersect an exon. Your result should be the same as the figure below.  

Bonus Question!

The Galaxy server is a highly malleable set of tools for any bioinformatician to utilize. Best of all, the pipeline process in data manipulation allows workflows to be saved and exported. This way your future colleagues will be able to upload the workflow to view your parameter and process, or run the same workflow. Just as it is a helpful to create workflows for others, you can create one for yourself as reference or to save locally. Galaxy is free to use, but the size is limited unless you create an account, which extends your storage to 250GB of storage space. However, you will soon find out that the limitations in storage space for processing bioinformatic data is a constant struggle. In this bonus question, we will create a Galaxy workflow based on the previous four questions. First, create an account and login. Be sure to double check that your history from questions one to four are still present in the history list on the right-hand column. That will be the history list you will create a workflow for. First the history list panel (right-hand column), click the cog-wheel icon for History Options and choose the option for Extract Workflow. See below for a guide on the processes. Next, create a name (as well as filename) for your workflow and ensure all the tools/data in both columns are checked. Lastly, click Create Workflow. On the top menu bar, click Workflow to open the page of your created workflows. You also have the option of exporting the workflow as a .ga file format specific for Galaxy.
